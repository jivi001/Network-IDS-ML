# Network Intrusion Detection System (NIDS-ML)

A production-grade, hybrid machine learning system designed for the detection of network intrusions and anomalous traffic behavior. This system utilizes a multi-tiered diagnostic cascade to achieve highly precision-calibrated alerts while preserving a robust defense against uncharacterized zero-day attacks.

## Architecture: 4-Tier Hybrid Cascade

The NIDS architecture relies on four consecutive operational tiers:

1. **Tier 1 - Known Attack Classification (Supervised Stacking Ensemble):** The primary diagnostic layer utilizes a stacking ensemble consisting of a `BalancedRandomForestClassifier`, `LGBMClassifier` (LightGBM), and a `CalibratedClassifierCV` built on a Support Vector Classifier (SVC). These models are cross-fitted and fed into a `LogisticRegression` meta-learner. Tier 1 generates probabilistic assessments predicting standard attack vectors (e.g., DoS, Probe, R2L, U2R). Traffic flagged as an attack with high confidence (default threshold: 0.90) immediately triggers a categorical alert.
2. **Tier 2 - Zero-Day Anomaly Detection (Unsupervised Fusion):** Traffic designated as "Normal" or below the confidence threshold by Tier 1 passes to Tier 2. This tier employs a customized fusion metric coupling a Variational Autoencoder (VAE) reconstruction error with `IsolationForest` decision paths. The VAE operates via a symmetric architectural reduction, embedding features into a lower-dimensional latent space. Distributions producing cumulative errors exceeding the 95th percentile trigger an anomaly alert, signaling possible zero-day behavior.
3. **Tier 3 - Interpretability Engine:** Alerts generated by Tier 1 and Tier 2 are independently verified using SHapley Additive exPlanations (`shap`). Using `TreeExplainer` algorithms, the system computes the exact predictive contribution of each network feature in real-time, thereby mapping internal model states to observable network metadata for Security Operations Center (SOC) review.
4. **Tier 4 - Adaptive Retraining & Drift Tracking:** Continuous streams of inference telemetry are actively monitored using ADaptive WINdowing (ADWIN) algorithms. If concept drift is documented within the error-rate distributions, structural alerts are appended to the deployment logging pipeline. Furthermore, an active learning module extracts the most statistically uncertain evaluations utilizing Shannon entropy coupled with K-Means diversity sampling, appending them to a persistent `FeedbackBuffer` designed to aggregate subsequent human-in-the-loop analyst feedback. Once capacity vectors are reached, the CI/CD pipeline triggers an automated model retraining protocol.

## Algorithmic Parameters

The parameters designated within the configuration files strictly follow optimization trials across varied network loads.
Key architectural parameters include:

- `StackingEnsemble`: Bootstrapping `n_estimators` mapped to 200, unbalance settings assigned exclusively to LGBM mechanisms.
- `VAEAnomalyDetector`: `latent_dim` calibrated to 16 parameters, binary cross-entropy reconstruction loss, beta scaled to 1.0.
- `FeatureSelector`: Executed utilizing Borda count rank aggregation spanning Random Forest Gini impurity scaling, SHAP matrix magnitude scoring, and Shannon Mutual Information indices, resolving the dimension boundary to the optimal subset (default 20 to 30 metrics).
- `ADWINDriftDetector`: Window variance magnitude limit `delta` set empirically to 0.002 to suppress micro-fluctuations.

## Training Pipeline and Evaluation

The system executes data formatting utilizing `NIDSPreprocessor`, converting continuous variables through `RobustScaler` methods (to mitigate unbounded packet sizes limits) whilst one-hot encoding discrete flags. Imbalanced training metrics are handled implicitly within the bootstrapping algorithms rather than dependent upon static SMOTE approximations.

Model verification prioritizes threat detection optimization:

- **MCC (Matthews Correlation Coefficient):** Defines absolute dataset classification efficiency, targeting values >0.92.
- **Alert Fatigue Index (AFI):** Defines constraints on FP metric reporting logic. Calculates FP accumulation against absolute alert limits, targeting indexes <0.05.
- **F2-Score:** Evaluates model capabilities via a statistically weighted standard that penalizes False Negatives twice as strictly as False Positives, targeting values >0.95.

## Deployment and CI/CD Integrations

Deployments are strictly orchestrated through automated Continuous Integration (CI) and Continuous Deployment (CD). The `.github/workflows/nids-mlops.yml` executes linting, dependency vulnerability scans, tests, and standard MLflow metric assertions before Docker container generation.
Inferences are exposed via a localized FastAPI REST runtime and WebSocket infrastructure, integrated with Prometheus metric scraping across specific distribution telemetry to enforce real-world visibility into systemic health and degradation limits.

## Project Usage

### 1. Environment and Requirements

Python 3.11 with virtual environment isolation is required to prevent standard dependency pollution.

```bash
git clone https://github.com/jivi001/Network-IDS-ML.git
cd Network-IDS-ML
python -m venv nids_env
source nids_env/bin/activate
pip install -r requirements.txt
pip install -e .
```

### 2. Training Instantiation

Datasets spanning `NSL-KDD` and `UNSW-NB15` are natively supported and expected to populate the `data/raw/` path.

```bash
python scripts/train.py --config configs/training/hardened_rf.yaml
```

### 3. Server Instantiation

Endpoints including `/predict`, `/predict/batch`, and `/explain` are routed utilizing an ASGI worker schema standard to asynchronous processing.

```bash
cd deployment
docker-compose up --build
```

Requests operate over TCP port `8000`. Detailed reference material surrounding the precise definitions of class parameters and programmatic access exists within `docs/API_REFERENCE.md`. All internal configurations are mutable directly via the `configs/` file structures.
